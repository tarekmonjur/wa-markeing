## এটা কী করে

ক্যাম্পেইন তৎক্ষণাৎ লঞ্চ না করে ভবিষ্যতে ডেলিভারির জন্য শিডিউল করা যায়। সিস্টেম BullMQ ডিলেইড জব ব্যবহার করে যা Redis এ সংরক্ষিত থাকে, তাই সার্ভার রিস্টার্ট হলেও শিডিউল বজায় থাকে।

## ধাপসমূহ

### ১. ক্যাম্পেইন তৈরি

```json
POST /api/v1/campaigns
{
  "name": "ঈদ মোবারক ২০২৬",
  "sessionId": "uuid",
  "templateId": "uuid",
  "groupId": "uuid"
}
```

### ২. শিডিউল সেট করুন

```json
POST /api/v1/campaigns/:id/schedule
{
  "scheduledAt": "2026-05-01T10:00:00.000Z",
  "timezone": "Asia/Dhaka"
}
```

### ৩. শিডিউল বাতিল

```
DELETE /api/v1/campaigns/:id/schedule
```

স্ট্যাটাস **DRAFT** এ ফিরে যায়।

### ৪. পুনরায় শিডিউল

```json
PATCH /api/v1/campaigns/:id/reschedule
{
  "scheduledAt": "2026-05-02T08:00:00.000Z"
}
```

## স্ট্যাটাস ফ্লো

```
DRAFT → SCHEDULED → RUNNING → COMPLETED
```

## রিকারিং ক্যাম্পেইন

| প্যাটার্ন | বিবরণ |
|----------|--------|
| **দৈনিক** | প্রতিদিন একই সময়ে রিপিট |
| **সাপ্তাহিক** | নির্বাচিত দিনে রিপিট (০=রবি, ৬=শনি) |
| **মাসিক** | নির্দিষ্ট তারিখে রিপিট (১–২৮) |

ঐচ্ছিক `endDate` দিয়ে রিকারিং ক্যাম্পেইন স্বয়ংক্রিয়ভাবে বন্ধ করা যায়।

## API এন্ডপয়েন্ট

```
POST   /api/v1/campaigns/:id/schedule    — শিডিউল করুন
PATCH  /api/v1/campaigns/:id/reschedule  — সময় পরিবর্তন
DELETE /api/v1/campaigns/:id/schedule    — শিডিউল বাতিল
```

## সমস্যা সমাধান

| সমস্যা | কারণ | সমাধান |
|--------|------|--------|
| `400 Bad Request` | সময় অতীতে | ভবিষ্যতের UTC সময় ব্যবহার করুন |
| শিডিউলে আটকে আছে | Redis বা BullMQ ওয়ার্কার বন্ধ | Redis সংযোগ পরীক্ষা করুন |
| সেশন বিচ্ছিন্ন | ফোন অফলাইন হয়ে গেছে | পুনরায় সংযোগ করুন |
